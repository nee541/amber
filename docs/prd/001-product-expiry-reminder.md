# 001 商品生产日期和保质期存储与提醒 PRD

## 1. 背景

Amber 是一个本地优先的物品档案管理应用。生产日期、保质期和到期提醒是物品管理中最基础也最常用的能力，适用于食品、药品、化妆品、家用品、收藏品等多种物品。

本功能的目标是在 MVP 阶段提供一个足够简单的手动录入流程，同时保留后续扩展空间，例如直接录入到期日期、系统通知、扫码、OCR、图片归档和更复杂的库存管理。

## 2. 目标

- 支持用户手动记录商品的生产日期和保质期。
- 根据生产日期和保质期自动计算到期日期。
- 在应用内展示商品的保质期状态和提醒信息。
- 支持全局提醒规则，并允许单个商品覆盖提醒设置。
- 支持按状态、分类、存放位置和关键词查看商品。
- 保持字段和日期模型可扩展，避免将 MVP 的录入方式写死。

## 3. 非目标

MVP 阶段暂不实现以下能力：

- 扫码识别商品信息。
- OCR 识别生产日期、保质期或包装文字。
- 拍照、票据、扫描件、PDF 等文件归档。
- 批量导入，包括表格或 CSV 导入。
- 云同步、账号、多设备协作或多人共享。
- 系统通知和后台提醒。
- 自动联网补全商品资料。

以上能力可作为后续 feature 进入独立 PRD。

## 4. 用户场景

### 4.1 快速记录食品

用户购买一批食品后，只想快速录入名称、分类、生产日期和保质期。应用自动计算到期日期，并在临期时出现在提醒列表中。

### 4.2 管理药品或化妆品

用户记录药品或化妆品的生产日期和保质期，希望在到期前较早收到提醒。某些商品需要单独设置更早的提醒时间。

### 4.3 查看临期和过期商品

用户打开应用后，可以直接看到即将过期和已经过期的商品，并决定使用、丢弃、归档或继续保留记录。

## 5. MVP 范围

### 5.1 商品录入

MVP 阶段只支持手动录入单个商品，不支持表格、CSV 或其他批量导入方式。

必填字段：

- 名称
- 分类
- 生产日期
- 保质期数值
- 保质期单位

可选字段：

- 数量
- 存放位置
- 备注
- 自定义提醒提前天数

分类规则：

- MVP 提供基础预设分类：食品、药品、化妆品、家用品、收藏品、其他。
- 预设分类作为初始数据存在，不作为不可变系统分类；用户可以重命名或删除预设分类。
- 用户可以自定义输入新的分类。
- 新增商品时，如果用户输入不存在的分类，系统自动创建该分类。
- 商品允许重名，依靠生产日期、分类、存放位置等信息区分不同记录。
- 分类支持重命名。
- 删除已被商品使用的分类时，需要先将关联商品迁移到其他分类。

存放位置规则：

- 存放位置使用可复用的位置管理。
- 商品允许不填写存放位置。
- 用户可以创建、编辑和选择常用位置，例如冰箱、冷冻室、厨房柜、药箱、卧室抽屉。
- 存放位置支持重命名。
- 删除已被商品使用的位置时，需要先将关联商品迁移到其他位置。

数量规则：

- MVP 阶段数量只记录简单数字。
- 数量为空表示未记录数量，不默认等同于 `1`。
- 暂不支持数量单位，例如盒、瓶、袋、片。

保质期单位支持：

- 天
- 月
- 年

### 5.2 日期计算

系统根据以下信息计算到期日期：

- 生产日期
- 保质期数值
- 保质期单位

MVP 阶段主要支持“生产日期 + 保质期”的录入方式。代码和数据模型需要预留扩展能力，以便后续支持：

- 直接录入到期日期
- 只知道年月的不完整日期
- 开封后保质期
- 多批次商品
- 其他日期来源

### 5.3 商品状态

商品状态分为三类：系统计算状态、用户操作状态和回收站状态。

系统计算状态：

- 正常
- 临期
- 已过期

用户操作状态：

- 在用或持有
- 已用完
- 已丢弃
- 已归档

回收站状态：

- 未删除
- 已删除

状态判断规则：

- 已过期：当前日期晚于到期日期。
- 临期：当前日期未超过到期日期，且距离到期日期的天数小于等于提醒提前天数。
- 到期当天显示为“今日到期”，仍属于临期；从到期日期之后的第一天开始显示为已过期。
- 正常：当前日期未超过到期日期，且未进入提醒窗口。
- 已用完、已丢弃、已归档的商品默认不进入活跃提醒列表，但仍可在历史或筛选结果中查看。
- 已用完、已丢弃、已归档的商品允许恢复为在用或持有。
- 用户删除商品后，商品进入回收站，不进入默认列表、提醒列表或常规搜索结果。

### 5.4 提醒规则

MVP 阶段只做应用内提醒。

提醒规则：

- 应用提供全局默认提醒提前天数。
- 商品可以设置自定义提醒提前天数。
- 商品自定义设置优先于全局默认设置。
- 未设置商品自定义提醒时，使用全局默认提醒规则。

默认提醒窗口：

- 全局默认：到期前 30 天。
- 可选快捷值：当天、提前 7 天、提前 30 天、提前 90 天。
- 允许用户输入自定义天数。
- 修改全局默认提醒提前天数后，未设置商品级自定义提醒的商品立即使用新的全局规则。
- 已设置商品级自定义提醒的商品不受全局默认提醒变更影响。

应用内提醒表现：

- 首页或提醒页展示“即将过期”和“已过期”商品。
- 商品列表中展示状态标签。
- 商品详情中展示到期日期、剩余天数和提醒规则来源。

系统通知作为后续 feature，不进入 MVP。

### 5.5 列表和查看

MVP 需要支持以下查看方式：

- 即将过期商品
- 已过期商品
- 按分类查看
- 按存放位置查看
- 按未设置位置查看
- 关键词搜索

列表展示字段：

- 名称
- 分类
- 到期日期
- 剩余天数或过期天数
- 状态
- 数量
- 存放位置

默认排序：

- 默认列表优先展示临期和已过期商品。
- 同一优先级内按到期日期从近到远排序。
- 已用完、已丢弃、已归档的商品不在默认列表和提醒列表展示，但可以通过状态筛选查看。
- 存放位置为空的商品在列表中显示为“未设置位置”。

### 5.6 商品详情

商品详情展示：

- 名称
- 分类
- 生产日期
- 保质期
- 自动计算的到期日期
- 当前状态
- 数量
- 存放位置
- 备注
- 提醒设置

详情页应允许编辑商品信息和用户操作状态。

创建时间和更新时间需要存储，但 MVP 界面不展示。

### 5.7 回收站

MVP 支持商品删除和回收站。

回收站规则：

- 删除商品时不做硬删除，而是移入回收站。
- 删除商品时保留原用户操作状态，并记录删除时间。
- 恢复商品时清除删除时间，商品回到删除前的用户操作状态。
- 回收站商品不进入默认商品列表、提醒列表、临期判断入口或常规搜索结果。
- 回收站支持永久删除商品记录。
- 永久删除需要二次确认。

## 6. 数据模型

### 6.1 Item

```ts
type ShelfLifeUnit = "day" | "month" | "year";

type ItemUserStatus = "active" | "used_up" | "discarded" | "archived";

type Item = {
  id: string;
  name: string;
  categoryId: string;
  productionDate: string;
  shelfLifeValue: number;
  shelfLifeUnit: ShelfLifeUnit;
  quantity?: number;
  storageLocationId?: string;
  note?: string;
  customReminderDays?: number;
  userStatus: ItemUserStatus;
  deletedAt?: string;
  createdAt: string;
  updatedAt: string;
};
```

### 6.2 预留扩展字段

为避免将 MVP 录入方式写死，后续可以引入日期来源字段：

```ts
type ExpiryDateSource = "production_date_and_shelf_life" | "direct_expiry_date";
```

后续也可以扩展为支持不完整日期：

```ts
type DatePrecision = "day" | "month" | "year";
```

MVP 可以先不暴露这些字段到 UI，但 domain 层的日期计算逻辑应避免散落在页面组件中。

## 7. 业务规则

- 保质期数值必须为正整数。
- MVP 阶段禁止生产日期晚于当前日期。
- 到期日期由 domain/helper 层统一计算。
- 生产日期和到期日期按本地日历日期处理，日期字段使用 `YYYY-MM-DD`，不包含时分秒。
- 按天计算保质期时，生产日期加保质期天数得到到期日期，例如 `2026-01-01 + 1天 = 2026-01-02`。
- 月和年保质期采用自然日期加法，例如 `2026-01-15 + 1个月 = 2026-02-15`，`2026-01-31 + 1个月 = 2026-02-28/29`。
- 商品临期状态由到期日期、当前日期和提醒提前天数共同决定。
- 用户操作状态为已用完、已丢弃或已归档时，不展示在默认提醒列表。
- `deletedAt` 不为空的商品只在回收站中展示。
- 已过期商品仍允许被标记为已用完、已丢弃或已归档。
- 已用完、已丢弃、已归档商品允许恢复为在用或持有。
- 同名商品允许同时存在，不做强制去重。
- 分类和存放位置被商品引用时不能直接删除，需要先完成迁移。
- 未被商品引用的预设分类允许删除。
- 编辑商品的生产日期、保质期数值或保质期单位后，需要立即重新计算到期日期和提醒状态。
- 搜索和筛选不应改变商品状态，只改变列表展示结果。

## 8. 页面与交互

### 8.1 商品列表页

核心能力：

- 查看全部活跃商品。
- 搜索商品名称。
- 按分类、存放位置和状态筛选。
- 按“未设置位置”筛选存放位置为空的商品。
- 通过状态筛选查看已用完、已丢弃和已归档商品。
- 快速进入新增商品和商品详情。

### 8.2 新增和编辑商品

核心能力：

- 输入必填字段。
- 输入可选字段。
- 自动预览到期日期。
- 自动预览当前状态。
- 设置使用全局提醒或自定义提醒。
- 新增商品时可以快速创建分类或存放位置。
- 编辑生产日期或保质期后，界面立即刷新到期日期和状态预览。

### 8.3 应用内提醒页

核心能力：

- 按“已过期”和“即将过期”两组查看提醒商品。
- 每组内部按到期日期从近到远排序。
- 进入商品详情。
- 将商品标记为已用完、已丢弃或已归档。

### 8.4 设置页

核心能力：

- 查看和修改全局默认提醒提前天数。
- 提供常用快捷值：当天、提前 7 天、提前 30 天、提前 90 天。
- 支持输入自定义提醒提前天数。
- 说明商品级自定义提醒会优先于全局默认提醒。
- 管理分类，包括创建、重命名、删除未被引用的分类，以及迁移后删除已被引用的分类。
- 管理存放位置，包括创建、重命名、删除未被引用的位置，以及迁移后删除已被引用的位置。
- 进入回收站，查看、恢复或永久删除已删除商品。

## 9. 验收标准

- 用户可以新增一个包含名称、分类、生产日期和保质期的商品。
- 系统可以正确计算并展示到期日期。
- 系统可以根据全局提醒规则判断商品临期状态。
- 单个商品可以覆盖全局提醒提前天数。
- 商品列表可以展示正常、临期和已过期状态。
- 提醒页可以展示即将过期和已过期商品。
- 用户可以将商品标记为已用完、已丢弃或已归档。
- 已用完、已丢弃和已归档商品不进入默认提醒列表。
- 支持按分类、存放位置、状态和关键词查看商品。
- 支持按未设置位置筛选商品。
- 支持基础预设分类，并允许用户自定义分类。
- 支持删除未被商品引用的预设分类。
- 新增商品时输入不存在的分类会自动创建分类。
- 支持可复用的位置管理。
- 商品存放位置允许为空。
- 商品存放位置为空时显示为“未设置位置”。
- 支持在设置页修改全局默认提醒提前天数。
- 支持在设置页管理分类和存放位置。
- 允许存在多个同名商品。
- 数量为空时展示为未记录数量。
- 商品列表默认按临期和已过期优先，并按到期日期从近到远排序。
- 编辑商品生产日期或保质期后，系统立即重新计算到期日期和提醒状态。
- 支持删除商品到回收站。
- 支持从回收站恢复商品。
- 支持从回收站永久删除商品，且永久删除前需要二次确认。
- 已用完、已丢弃和已归档商品可以恢复为在用或持有。
- 创建时间和更新时间会被存储，但 MVP 界面不展示。
- 日期计算逻辑位于独立 domain/helper 模块，不直接写在 React 组件中。
